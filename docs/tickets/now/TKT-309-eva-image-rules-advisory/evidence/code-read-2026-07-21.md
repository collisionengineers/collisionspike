# TKT-309 — code-read evidence, 2026-07-21

Read-only inspection of `packages/domain/src/contracts/image-rules.ts` and `case-status.ts` at
the time of the inbound-triage rebuild investigation.

`image-rules.ts:19-28` (pre-change) carried an operator ruling dated the same day, 2026-07-21
(TKT-130): "THOSE THREE RULES ARE THE WHOLE CONTRACT... Do not reintroduce" a fourth
(auto-exclude-review) gate. TKT-130's own acceptance lines require the three rules to gate
Review/readiness ("At least one accepted usable image set is required... zero accepted images
is Not Ready"), and `canonical-readiness.test.ts` pinned that with parametrized cases (all
images excluded, missing overview, missing damage-closeup — each asserted `missing_images` /
`not-ready`).

A further, same-day operator ruling supersedes this: the three rules themselves become advisory.
Confirmed directly with the operator before implementing, given the concrete consequence spelled
out — a case with zero accepted images can now reach `ready_for_eva` and be submitted to EVA,
with the image gap shown only as advisory checklist copy.

## Blast radius traced before implementing

- `evaluateCaseReadiness` (`case-status.ts`) is the SINGLE canonical readiness evaluator —
  `readinessForCase` and `canSubmitCaseToEva` (`case-readiness.ts`) are thin wrappers around it,
  not a parallel implementation. One change point.
- `statusForReviewCase`'s decision tree assigns `missing_images` at step 3
  (`fieldContractValid && !baseImagesValid`) — this becomes unreachable once images no longer
  gate `ready`, since step 2 (`readiness.ready -> ready_for_eva`) now fires first whenever the
  field/address/vehicle/source contract is met.
- Six tests across three files pinned the old blocking behaviour and needed updating to the new
  intended behaviour (not just deleting): `canonical-readiness.test.ts` (3), `case-status.test.ts`
  (3), plus one data-api fixture asserting the old `missing_images` status code (100000004)
  where the new expectation is `ready_for_eva` (100000007).
