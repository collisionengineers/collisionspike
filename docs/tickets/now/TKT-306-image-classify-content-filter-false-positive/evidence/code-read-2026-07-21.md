## Why this is a false positive on every healthy call

Azure OpenAI's `Microsoft.DefaultV2` RAI policy (the policy on the gpt-5 deployment this repo
uses) annotates **every** successful response — not only refused ones — with
`content_filter_results` (per-choice) and `prompt_filter_results` (per-prompt) objects, each key
named for a harm category (`hate`, `self_harm`, `sexual`, `violence`, `jailbreak`, ...) with a
`severity`/`filtered`/`detected` field. `JSON.stringify` on any such response therefore always
contains the substring `content_filter`, so the regex fallback matched 100% of calls under this
policy, independent of `finish_reason`.

`services/orchestration/src/adapters/aoai.ts:306`, the sibling caller against the same provider,
checks only `finish_reason === 'content_filter'` and has no such fallback — it was unaffected.

## Test-fixture gap

Every success fixture in `image-classify.test.ts` prior to this ticket (e.g. the `valid` object
used across `imageClassificationOutcomeFromResponse` tests) was annotation-free — no
`content_filter_results`/`prompt_filter_results` key — which is exactly why the existing suite
was green while the fallback discarded every real classification in production.
