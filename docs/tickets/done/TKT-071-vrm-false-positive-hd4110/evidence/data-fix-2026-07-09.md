## TKT-085 root cause (established, not assumed)

- "OCTOBER" appears NOWHERE in the source email (raw .eml grep = zero hits) and the
  current engine's sniff of the same message returns the real plate `BE57JDS`.
- The live value entered through the **/parse VRM path**: `caseResolve` stamps the
  parser-derived VRM onto BOTH `case_.vrm` and `inbound_email.body_vrm`
  (`api/src/functions/internal.ts` "upgrade its body_vrm to the best VRM"). The live
  FC1 parse of the attached Audatex PDF surfaced the month word; a local re-parse of the
  same PDF with engine-v2.10 yields `BE57JDS` (correct).
- The accepting hole (now closed, engine-v2.10 + TS filter): NO layer rejected an
  all-alphabetic "mark" — the validation regex (`[A-Z0-9]{2,8}`) passes OCTOBER, provider
  label rules carry no digit requirement, and `_is_suspicious_value` had no all-alpha
  guard. The wave adds: all-alpha rejection + month/day denylist + function-word-head
  rejection in `vrm_candidate_is_bad`/`_is_suspicious_value` (Python) and the mirrored
  denylists + proximity/tight anchoring in `packages/domain/src/domain/vrm-filter.ts` (TS).
