# focus-flow — WCAG-AA accessibility audit (static)

**Scope:** static source audit of `index.html`, `queues.html`, `case-detail.html` in
`directions-r2/focus-flow/`. Method: read the rendered markup + CSS, compute contrast from the
frozen token palette, trace keyboard/ARIA/focus/tap-target behaviour through the inline JS.
**Auditor:** accessibility-engineer · **Date:** 2026-06-25

**Verdict: GATED — not convergence-ready.** Strong a11y floor (focus visible everywhere,
reduced-motion handled properly, shape+colour status coding, landmarks) but it gates on a small set
of concrete **Level A** failures: the primary grid rows and the photo-exclude switch are
mouse-only, and the case-detail form fields have no programmatic labels. Fix those plus the
`ink-faint` contrast and this is a 4/5 direction.

**Score: 2.5 / 5**

---

## What passes (credit where due)

- **Visible focus on everything.** Global `*:focus-visible{outline:2px solid var(--accent);
  outline-offset:2px}`. Accent `#8A2E55` measures **7.67:1 vs the card** and **6.26:1 vs the stage
  bg** — well above the 3:1 non-text floor. This is the single strongest thing in the build.
- **Reduced motion is genuinely handled** — not just a token gesture. All three files carry
  `@media (prefers-reduced-motion:reduce)`, and `case-detail.html` additionally guards the JS
  "advance" animation with `window.matchMedia('(prefers-reduced-motion: reduce)')` before running the
  rise/translate. The motion path has a real reduced fallback.
- **Status is shape-coded AND colour-coded** — `●` review / `◆` held / `✓` ready / `○` new /
  `△` duplicate. Never colour-alone.
- **Landmarks + named regions.** `<nav aria-label="Primary">`, `<aside aria-label="…queue">`,
  `<main>`, mobile `<nav aria-label="Mobile">`. The pipeline spines carry `role="img"` + a
  descriptive `aria-label` so the numeric glyphs get a text alternative.
- **Rail rows are real `<button>`s** (index + case-detail) — focusable, native Enter/Space, J/K
  nav, `min-height:44px` touch target.
- **Most contrast is comfortably AA.** `--ink` 14.4:1, `--ink-secondary` 7.6:1, accent 7.7:1, and
  the status text-on-tint pairs all pass: review `#8e2c22`/tint **6.6:1**, held `#6f4b0d`/tint
  **6.5:1**, ready `#355028`/tint pass. **Amber is dark-on-amber** (`#8C6011`, no white-on-amber);
  **blocker pills** render `#8e2c22` on `#F6E2DC` at **6.6:1**. CE-brand rules respected.
- Icon-only controls that exist (refresh `↻`, more `⋯`, nowstrip arrows) have `aria-label`s;
  photo-role `<select>`s and address radios are labelled.

## Failures (located, by severity)

### Critical — Level A keyboard / name-role-value

| # | File | Location | Issue |
|---|------|----------|-------|
| C1 | queues.html | `<tr onclick="location='case-detail.html'">` (lines 249–320) | **Grid rows are mouse-only.** `<tr>` has no `tabindex`, no `role`, no `keydown` and is not focusable, so the global focus-visible never applies and there is no per-row keyboard activation. The only keyboard path is the global J/K + Enter, which (a) requires focus to be outside any input and (b) always navigates to `case-detail.html` regardless of the selected row. Opening a case — the screen's primary action — fails **2.1.1 Keyboard**. |
| C2 | case-detail.html | `.sw` exclude switches (lines 440–443) | **`role="switch"` on a non-focusable `<span>`.** `aria-checked` is toggled in JS but the element has no `tabindex`, so a keyboard user cannot toggle photo exclusion — a real domain action ("any photo showing a person's reflection is unusable"). Fails 2.1.1 and **4.1.2**. Same pattern on the `.sw` in the order list. |
| C3 | case-detail.html | `.rditem.no` (lines 514, 519, 525) and `.mbar .lnk` (line 373) | Readiness deep-links are `<div onclick>` / `<span onclick>` with no `tabindex`/`role` — the "jump to the failing field" affordances (claimant email, overview photo, VAT conflict) are mouse-only. Fails 2.1.1. |

### Serious

| # | File | Location | Issue |
|---|------|----------|-------|
| S1 | case-detail.html | Fields tab, all 12 `.fieldrow` (lines 392–412) | **Form fields have no programmatic label.** The visible `.lbl` is a `<span>` — not a `<label for>`, not wrapping, no `aria-label`/`aria-labelledby`. Screen readers announce "edit text" with no name for every field. `.errmsg`/`.conflictmsg` are also not tied via `aria-describedby` (the email input sets `aria-invalid` but the error text is orphaned). Fails **1.3.1 / 4.1.2 / 3.3.2**. |
| S2 | all three | `--ink-faint #A99F8E` as informative text | **Contrast ~2.0–2.5:1, fails 1.4.3.** On the card surface it is **2.48:1**; on the stage/rail bg **2.03:1**. Used for genuinely meaningful small text: side-panel group labels ("Required fields"/"Evidence"), `.out small` "+1 more", `.regbadge.no` "— no reg", `.chips .cl` "Reason", nav `.navdiv` + `.foot`, the `.pos .of` "/14", and the `⌘K`/shortcut hints. Owner: ui-visual-designer. |

### Moderate

| # | File | Location | Issue |
|---|------|----------|-------|
| M1 | all three | `--ink-muted #6F6557` and `--st-held #8C6011` on the **stage/rail** bg `#E8E2D7` | ink-muted lands **4.43:1** and amber-due **4.29:1** on the `#E8E2D7` rail/topbar — just **under** the 4.5:1 floor for the small (11–13px) text there (provider name, due chips, "Updated 09:42", railhead label). Same tokens on the `#FBF9F4` card are fine (5.4 / 5.3:1). Either darken the tokens or sit this text on a card surface. |
| M2 | queues.html / case-detail.html | `.ptabs` + `.tabs` (queues 216–222; detail 378–386) | **Incomplete tab semantics.** Only some `.ptab`s carry `role="tab"`; `aria-selected` is set on one tab and never updated; there is no `role="tabpanel"`, no `aria-controls`, no arrow-key roving tabindex. The disabled "Enrichment" tab uses `pointer-events:none` with no `disabled`/`aria-disabled`. Functions via `<button>` click, but the ARIA is misleading. Fails 4.1.2 partially. |
| M3 | queues.html / index.html | `.chip` facet toggles (queues 236–239); `.acchead` accordions (index 296–311) | Toggle state not exposed: chips need `aria-pressed`; accordion headers need `aria-expanded` (+ `aria-controls`). State is conveyed by colour/twisty only. |
| M4 | case-detail.html | document head/header | **No `<h1>`.** The case workspace has no page heading — the first heading in the DOM is the modal's `<h2>`. Section labels are `.kick` `<div>`s, not headings, so heading-navigation is empty. (index.html does have an `<h1>` + `aria-labelledby` sections; queues.html has its `<h1>`.) |

### Minor

- Command palette (`.cmdk`) and submit `.modal`: `role="dialog"`/`aria-modal` present on the modal,
  but neither traps focus, moves focus reliably into the dialog on open, nor makes the background
  `inert`. Esc-to-close works.
- `.toast` "Submitted…" is not `aria-live`, so the submit confirmation is silent to SR users.
- queues `.sbar` spark buttons name themselves via `title` only (no visible text/`aria-label`) and
  encode queue type by dot colour alone (`rv`/`hd`/`rd`) — borderline.
- Address radios share the label "pick" ×3 (line 463–465); they should be labelled by the address
  text. `.mini` action buttons (index) are ~22–23px tall — just under the 24px AA target floor
  (2.5.8); rail/nav/btn targets are fine.

## Routing

- **ui-visual-designer** — S2 + M1: retune `--ink-faint` and the `--ink-muted`/`--st-held` usage on
  the `#E8E2D7` stage/rail so informative small text clears 4.5:1.
- **stitch-prototyper** — C1, C2, C3, S1, M2, M3, M4 + minors: make grid rows and the readiness/
  exclude controls keyboard-operable (real `<button>`/`<a>` or `tabindex`+`role`+`keydown`),
  associate field labels, complete the tab/toggle ARIA, add the case-detail `<h1>`.
- **mobile-ux-designer** — `.mini` and `.sw` hit-target sizing.

## Gating verdict
**BLOCK for convergence** until C1–C3 and S1 are resolved (Level A keyboard + name/role/value).
S2/M1 (contrast) must clear before any production port. The floor — focus, reduced motion, shape+
colour coding, landmarks, CE-compliant amber/blocker colours — is already in good shape.
