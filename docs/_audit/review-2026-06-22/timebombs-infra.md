# Ticking time bombs — config, secrets, gates, deploy (#7)

This section captures confirmed infrastructure-and-config "time bombs" — latent regressions that lie dormant until a routine action (a solution re-import, a bicep redeploy, an inbox activation, a stray `git add`) silently reverts hard-won live state. Each finding below was independently re-verified against the live Dev environment and the repo; several originally-higher severities were downgraded after the verifier checked them against documented Power Platform behavior and existing in-repo mitigations. Four findings were confirmed (with severity corrections noted inline); three were dropped during verification.

Confirmed findings by corrected severity: **0 critical · 0 high · 1 medium · 3 low** (4 total). Note: as originally graded the set was 1 critical / 1 high / 1 medium / 1 low, but every finding above informational was downgraded on verification — see each block's verifier note and the corrected severity in its heading.

## [MEDIUM] Live enrichment Function holds DVSA/DVLA credentials as plain-text app settings, and a bicep redeploy would reset them to KV refs against an empty vault

*(Originally filed "high"; downgraded to medium on verification — see note.)*

**Location:** Live `cespkenrich-fn-gi62sd` app settings `DVSA_CLIENT_SECRET` / `DVSA_API_KEY` / `DVLA_API_KEY` (literal values, not KV refs); `functions/enrichment/README.md:121-131`; `functions/enrichment/infra/main.bicep` (KV refs + Key Vault Secrets User grant, :222-258); KV `cespkenrichkvgi62sd` empty.

**Evidence:** Live probe `az functionapp config appsettings list -g rg-collisionspike-dev -n cespkenrich-fn-gi62sd` shows `DVSA_CLIENT_ID/DVSA_CLIENT_SECRET/DVSA_API_KEY/DVLA_API_KEY` all with `isKvRef:false` and literal values (e.g. `DVSA_CLIENT_SECRET=aBX8Q~_3CJB7...`), not `@Microsoft.KeyVault(SecretUri=...)` refs. `az keyvault secret list --vault-name cespkenrichkvgi62sd` returns `[]`. This contradicts the README §"Secret handling — Key Vault only" (lines 121-131, "values are Key Vault references ... resolved by ... managed identity") and `infra/main.bicep` (KV-reference app settings :222-237 + Key Vault Secrets User grant :245-258). A bicep deployment did run (`enrichment-infra-20260618`), so the live literals were manually written over the bicep KV-ref settings post-deploy. Two latent failures follow: (a) secrets sit in cleartext in app settings, readable by anyone with Reader/Contributor on the app or the deployment history; (b) a clean redeploy from bicep would replace the working plain values with KV refs pointing at the **empty** vault, so the token exchange fails and enrichment breaks at the next IaC deploy.

**Recommendation:** Inject the four DVSA/DVLA secrets into KV `cespkenrichkvgi62sd` and switch the live app settings to `@Microsoft.KeyVault(SecretUri=...)` refs (matching bicep), confirming the system-assigned MI has Key Vault Secrets User. This both closes the exposure and makes a bicep redeploy non-destructive. Secret rotation is optional given the accepted-non-sensitive stance.

**Verifier note (confidence: high):** Factual core verified live — the four settings are plain-text literals and the vault is empty, contradicting `main.bicep` and the README. Downgraded high→medium because the cleartext-exposure framing is overstated and not novel: the operator has explicitly accepted that the DVSA/DVLA keys are not sensitive (`docs/gated.md:194-196`), the deviation is already documented as a known "hygiene deviation" (`docs/architecture/live-environment.md:26`), and `DVSA_CLIENT_ID` is a non-secret GUID. The legitimately actionable, under-disclosed part is the IaC time bomb — a bicep redeploy resets the settings to KV refs against the empty vault and breaks enrichment until secrets are injected. Primarily a maintainability/operability fix.

## [LOW] Solution re-import will regress live intake flow wiring — repo `intake.definition.json` trails the live enrich + auto-merge chain

*(Originally filed "critical" as a two-claim finding; the env-var half was refuted and dropped, and the surviving flow-wiring half was downgraded to ~medium/low — see note.)*

**Location:** `flows/definitions/intake.definition.json` (3 child-flow invocations: `CS_Classify_Persist`, `CS_Parse`, `CS_Status_Evaluate`; no `Run_enrich` / `Run_case_resolve`); `flows/flow-state.json:56`; live `CS Intake` (hand-wired in the designer to call `Run_enrich` + `Run_case_resolve`, verified live 2026-06-21).

**Evidence:** `intake.definition.json` contains exactly three `workflowReferenceName` child-flow invocations — `CS_Classify_Persist` (line 336), `CS_Parse` (line 351), `CS_Status_Evaluate` (line 453) — and **no** `Run_enrich` / `Run_case_resolve` / `CS_Enrich` / `CS_Case_Resolve` invocation. The `Run_case_resolve` grep hits are incidental prose in two comments; the definition's own line-463 comment states: "WARNING — LIVE DIVERGES (verified 2026-06-21): the live CS Intake WAS subsequently wired to call Run_case_resolve (+ Run_enrich) after parse... THIS repo definition trails live — reconcile it ... before any solution re-import, else a re-import would regress the live wiring." `flows/flow-state.json:56` corroborates ("the repo intake.definition.json trails live ... reconcile before a solution re-import; live is authoritative"). Flow definitions are replaced on managed-solution import, so re-importing intake from the repo would drop the live enrich + auto-merge calls with no error surfaced.

**Recommendation:** Reconcile `flows/definitions/intake.definition.json` (and enrich/case-resolve) to the live wiring **now**, before any future solution export/import, so a re-import cannot regress the chain. Add a post-import verification step to DEPLOY-RUNBOOK that re-checks the live flow chain. (The original second recommendation — guarding env-var `currentValue` overrides on import — is unnecessary; see Dropped appendix.)

**Verifier note (confidence: high):** The flow-wiring claim is confirmed real — the three-child-call definition trails the live CS Intake, and a managed-solution re-import would regress the live enrich + auto-merge wiring. However it is already explicitly documented in-repo (the definition's own line-463 comment, `flow-state.json:56`, and MEMORY `intake-repo-trails-live`), and this is a Dev sandbox where a re-import is a deliberate operator action, not an automatic event — so ~medium, trending low, not critical. The finding's bundled second claim (a solution re-import "silently turning enrichment OFF" by resetting the env-var `currentValue` to its `defaultValue=false`) was refuted: per Microsoft Learn, a solution import does not overwrite an existing current value unless the imported solution carries one, and this manifest leaves `currentValue` null by convention (`environment-variables.json:11`), so the Dev `currentValue=true` survives a normal re-import. Note: the location header lists `flows/flow-state.json:56` under the `flows/` root (correct), not under `flows/definitions/`.

## [LOW] `MinIntakeDate` guard ships a hardcoded past date (2026-06-17) as the default — a not-fail-safe default for inboxes activated later

*(Originally filed "medium"; downgraded to low on verification — see note.)*

**Location:** `flows/definitions/intake.definition.json:13-16` (`MinIntakeDate` defaultValue `"2026-06-17"`); `flows/definitions/intake-shared-mailbox.definition.json:13-16`; `flows/flow-state.json:19,33`.

**Evidence:** Both intake definitions hardcode `"defaultValue": "2026-06-17"` and the metadata says "Set per-environment at activation = that inbox's go-live date." The guard `Drop_if_before_min_date` (`intake.definition.json:42-73`) drops mail with `receivedDateTime < MinIntakeDate` via Terminate Succeeded. Because the default is a fixed past date, an inbox activated without the operator explicitly resetting `MinIntakeDate` keeps a stale past cutoff — a weaker default than a fail-safe one. Nothing enforces the per-inbox reset, so the stale default is a (bounded) footgun.

**Recommendation:** Either compute the cutoff at activation (default to `utcNow()` so an un-set guard ingests nothing rather than a backlog), or make the parameter required with no default so import forces the operator to supply it.

**Verifier note (confidence: high):** Confirmed both defs hardcode the past `defaultValue` with nothing forcing a per-inbox reset, and the valid kernel — a fail-safe default would be a genuine improvement — holds. Downgraded to low because the finding is overstated on two counts: (1) magnitude — the triggers are Office 365 webhook/poll triggers (`OnNewEmailV3` / `SharedMailboxOnNewEmailV2`) that fire on newly-arriving mail once armed and do **not** retroactively sweep the existing mailbox, so worst-case exposure is the bounded window between the stale cutoff and actual go-live, not "months/years of backlog"; and (2) the recommendation to add a "set `MinIntakeDate` = go-live date" checklist item is **already done** — `docs/activation/multi-inbox-activation.md` step A.3 (lines 116-119), B.3 (lines 163-164), and Verify step 7 (lines 190-191) explicitly instruct setting it and test the guard.

## Dropped during verification

No Box MCP server is configured in this environment (empty `mcpServers`, no `mcp__box*` tools, plugin declares none) and the live Box tooling runs as Node/CLI via Bash, which the guard covers. The receiver (`functions/box-webhook/function_app.py:263`) is **not** gated by `BOX_API_ENABLED`, so the stated cause is wrong. The real (and already-mitigated) hazard — subscribing the webhook before KV signature keys are injected — is already sequenced in `REMAINING-STEPS.md:54-60` (secret injection + subscription before the gate flip). Recommending a `BOX_CLIENT_ID`-empty check is mistargeted (the receiver never reads it). At most a low belt-and-braces startup guard.
