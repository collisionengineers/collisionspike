# Provider / inspection address-matching logic — overlap & correctness (#9)

This review covers the provider sender-domain match and the inspection-address matcher, looking for correctness defects and duplicated/drifting logic across the TypeScript domain code, the Power Automate flow definitions, and the `functions/addressmatch` Python module. The core safety properties hold (ambiguous provider domains never auto-pick per ADR-0011; no branch emits the "Image Based Assessment" literal without a non-empty reason), but a genuine cross-district false-match bug exists in `district_matches`, the full-postcode resolve branch skips the corpus lookup the plan prescribes, and the canonical provider-match rule is hand-synced across multiple copies with no byte-identity guard.

Confirmed findings by severity: 1 high, 2 medium, 1 low, 1 info (5 total).

## [HIGH] district_matches() startswith logic contradicts its own docstring + the plan's verification — B5 spuriously swallows B50 (untested cross-district false match)

**Location:** `functions/addressmatch/postcode.py:109-127` (`district_matches`); claim contradicted at `postcode.py:115-117` and `docs/plans/phase-4-address-and-chaser/inspection-address-matching.md:198-201`; untested in `functions/addressmatch/tests/test_addressmatch.py:99-111`.

**Evidence:** The function returns `cand_out.startswith(co)` (`postcode.py:127`). Its docstring asserts: "`startswith` (not equality) so a 2-char district like `B5` does not spuriously swallow `B50`" (`postcode.py:115-117`). The plan's verification section asserts the same expected behaviour: "`district_matches('B5','B50 1AA')==False`" (`inspection-address-matching.md:198`). Running the live code: `district_matches('B5','B50 1AA')` returns **True** (and `district_matches('CH5','CH50 1AA')`→True, `district_matches('M1','M12 6BD')`→True). So a Case whose Loc district is `B5` would match a corpus yard in the genuinely-different district `B50` — a cross-district false positive that could auto-fill the WRONG inspection address. The test suite only covers `CH5`/`CH41` (`test_addressmatch.py:99-111`), which do NOT collide because neither is a prefix of the other, so the bug is masked. The source corpus analysis this module claims to mirror defines the rule as district **equality**, not prefix: "Loc is a PART postcode whose district == a repairer's district" (`raw/principalandrepairersheets/outputs/_scripts/task4.py:14,169`).

**Recommendation:** Either (a) change `district_matches` to outward-code **equality** (`cand_out == co`) to match the corpus analysis and the documented intent, or (b) if a deliberate sector→district widening is truly wanted, fix the docstring + plan to stop claiming B5≠B50 and constrain the prefix so it only matches on full-token boundaries. Add the exact `district_matches('B5','B50 1AA')` and `('M1','M12 6BD')` assertions the plan already specifies to the test suite.

**Verifier note (confidence: high):** CONFIRMED by executing the live code. `functions/addressmatch/postcode.py:127` returns `cand_out.startswith(co)`. Ran it: `district_matches('B5','B50 1AA')=True`, `('CH5','CH50 1AA')=True`, `('M1','M12 6BD')=True`. This directly contradicts (a) the function's own docstring (`postcode.py:115-118`: "startswith (not equality) so a 2-char district like B5 does not spuriously swallow B50"), and (b) the plan's verification (`inspection-address-matching.md:198`: "district_matches('B5','B50 1AA')==False"). The corpus analysis it mirrors defines the rule as district EQUALITY, not prefix (`task4.py:14,165,169-172`). The test suite (`test_addressmatch.py:99-111`) only uses CH5/CH41/M12 — no prefix-collision pair — so the bug is genuinely untested/masked, exactly as claimed. Root cause: `'B50'.startswith('B5')` is True at the string level; the right test is outward-code equality. Not intentional-by-design — the docstring and plan both demand the opposite of the actual behaviour.

**Verifier correction:** Severity is closer to MEDIUM than HIGH. The false candidate is admitted (`matching.py:140` gates on `district_matches`), but two downstream factors limit blast radius: (1) `matching.py:152` computes `exactDistrict = (cand_out == case_outward)` and the sort at `matching.py:168-176` ranks exact-district candidates ABOVE startswith-only ones, so a spurious B50 match only becomes the top recommendation when NO genuine B5-district site is also a candidate; (2) per `README:40-46` the literal address is never silently written — a reviewer decision with a non-empty reason is required. So it surfaces a WRONG recommendation to a human reviewer, not a silent auto-fill. The bug, docstring/plan contradiction, divergence from the corpus equality rule, and the test-masking are all exactly as reported. Note also the upstream Dataverse `$filter` is the same flawed `startswith(cr1bd_postcode,'<outward>')` (`README:51-54`), so the defect is consistent end-to-end, not corrected upstream. Fix: change `postcode.py:127` to `cand_out == co` and add the B5/B50 and M1/M12 assertions the plan already specifies.

## [MEDIUM] Full-postcode Loc branch never consults the corpus — emits a postcode-only 'address' yet marks it confirmed_physical + matched, diverging from the plan

**Location:** `functions/addressmatch/matching.py:318-341` (the `parsed.kind == 'full'` branch in `resolve`); spec it violates: `docs/plans/phase-4-address-and-chaser/inspection-address-matching.md:86-91` (§3 step 1).

**Evidence:** When Loc is a full postcode the branch builds `chosen` with `addressLines: []` (`matching.py:320-328`) and serialises only the postcode, returning `decisionMode=confirmed_physical, matched=True`. Verified: `resolve(case_loc='OL1 3QR', repairers=[<a repairer with full address lines AND postcode OL1 3QR>])` yields inspectionAddress `'OL1 3QR\n\n\n\n\n'` — the supplied corpus repairer's street/town are ignored; only the bare postcode survives, on a 6-line field that is otherwise blank. The plan's §3 step 1 explicitly requires: full → "resolve the address (corpus Repairer/InspectionAddress with that postcode, else postcode.io to validate + label) → confirmed_physical candidate" (`inspection-address-matching.md:88-90`). The code does neither corpus lookup nor postcode.io address-line enrichment (`normalize_with_postcodeio` only replaces the postcode string, `matching.py:232-240`). A postcode-only string labelled `confirmed_physical`/`matched=True` will pass the readiness gate without surfacing for review, despite having no street or town for the EVA field-9 6-line address (`case.json:63` expects a serialized 6-line address).

**Recommendation:** In the full-postcode branch, first search the supplied `repairers`/`inspection_addresses` for an exact-postcode corpus row and use its address lines; if none is found, do NOT claim `confirmed_physical`/`matched=True` on a bare postcode — return candidates / needsReviewerDecision (or `decisionMode=manual` flagged for review) so a street-less address is not silently accepted into EVA.

**Verifier note (confidence: high):** CORE FACTS CONFIRMED, but the finding is OVERSTATED (mislabels an intentional-by-design choice as a spec violation, and asserts an unproven live consequence). Verified in repo:

(1) `matching.py:319-341` — the `parsed.kind == 'full'` branch builds `chosen` with `addressLines: []` (line 322), never reads `repairers`/`inspection_addresses`, runs only `normalize_with_postcodeio` (which replaces the postcode string alone — `matching.py:232-240`), and returns `decisionMode=confirmed_physical`, `matched=True` (lines 335-339). `rank_candidates` also structurally returns `[]` for a full postcode (`matching.py:129-130`). So the corpus is genuinely never consulted on this path.

(2) The serialized output is exactly as the finding says: `serialize_six_lines([], "OL1 3QR")` = postcode line + 5 blanks = `"OL1 3QR\n\n\n\n\n"` (`postcode.py:130-156`). `case.json:63` confirms field 9 wants a 6-line address.

(3) There IS a real code-vs-plan divergence: `inspection-address-matching.md:86-90` (§3 step 1) prescribes full → "resolve the address (corpus Repairer/InspectionAddress with that postcode, else postcode.io ...)"; the code does neither corpus step.

WHERE THE FINDING IS WRONG / OVERSTATED:

(a) NOT an accidental bug — it is intended-by-design and explicitly locked by a passing unit test. `test_resolve_full_postcode_is_confirmed_physical` (`test_addressmatch.py:248-259`) passes EMPTY corpus lists and ASSERTS `decisionMode=="confirmed_physical"`, `matched is True`, and `inspectionAddress.split("\n")[0]=="OL1 3QR"`. The `resolve()` docstring (`matching.py:272-276`) states a full Loc "needs no district match; it is normalised and returned as a confirmed physical address." So the implementer deliberately chose not to enrich on this branch. The plan header itself (`inspection-address-matching.md:7-8`) says "the live route/contract is authoritative where it differs" — i.e. code is allowed to diverge from this planning runbook. Framing it as "violates the spec" is too strong.

(b) The "passes the readiness gate without surfacing for review" consequence is SPECULATIVE and likely false on the live path. The matcher Function (`cespkaddr-fn-i7m4re`) is deployed but NOT wired into the live pipeline — the address-resolve flow step is [RESERVED-FOR-USER] / operator-gated (`live-environment.md:27`; plan §1: "what remains is operator-gated: ... wiring the live address-resolve pipeline step", runbook step 6). The live Code App M1 gate is `mockup-app/src/domain/address-policy.ts`, which consumes a `hasPhysicalAddress` boolean (not the matcher string) and under the default `prefer_address` policy returns `decisionMode:'manual'` (line 118), never `confirmed_physical`. The finding cites no readiness-gate code proving a postcode-only field-9 is silently accepted.

(c) Severity "medium" is too high → low. Full postcodes in Loc are the minority (the corpus is ~57% part-postcode; a full unit postcode already pinpoints a real location, unlike a bare district), the branch is not live-pipeline-wired, and a unit-postcode-only field-9 is a thin-but-valid physical address rather than corrupt data. This is a quality/plan-reconciliation gap, not a defect that silently corrupts EVA submissions.

**Verifier correction:** The full-postcode branch (`matching.py:319-341`) does skip the corpus/postcode.io address-line lookup that the planning runbook (§3 step 1) describes, emitting a postcode-only "confirmed_physical"/matched field-9 string. This is real and worth reconciling, BUT it is intended-by-design (explicitly asserted by passing test `test_addressmatch.py:248-259` and documented in the `resolve()` docstring), the planning doc is non-authoritative where code diverges (its own header, lines 7-8), and the asserted downstream effect ("passes the readiness gate without surfacing for review") is unproven — the matcher is not yet wired into the live pipeline and the live Code App gate (`address-policy.ts`) does not consume this string. Reclassify as low-severity plan-vs-code reconciliation / minor quality gap, not a medium spec-violation defect. The recommendation (look up the corpus row by exact full postcode for address lines before claiming confirmed_physical) is still a reasonable improvement.

## [MEDIUM] Provider sender-domain match logic is triplicated (TS + standalone flow + inline intake) and must be kept byte-identical by hand

**Location:** `mockup-app/src/domain/provider-match.ts:51-103`; `flows/definitions/provider-match.definition.json:69-77` (`Filter_exact_domain`); `flows/definitions/intake.definition.json:193-200` (inline `Filter_exact_domain`).

**Evidence:** The same anchored exact-membership rule exists in three places. The standalone flow's own comment states it is duplicated and currently dead: "intake currently does the match inline and does NOT call this child, so it is orphaned-by-design" (`provider-match.definition.json:4`). The intake inline copy carries the same `@contains(split(replace(replace(toLower(...),'%0D',''),' ',''),'%0A'), senderDomain)` expression as the standalone flow (`intake.definition.json:197` == `provider-match.definition.json:73`), and both comment that they 'mirror provider-match.ts' (`intake.definition.json:190`). This is an intentional design (hot-path inline + ALM-reuse child) and the rule itself is correct, but three hand-synced copies of the canonical match is a drift hazard: a future fix to one (e.g. the CRLF/space-strip) can silently desync the others.

**Recommendation:** Keep, but add a CI/linter check (extend `flows/validate-flows.mjs`) that asserts the `Filter_exact_domain` `where` expression is byte-identical between `intake.definition.json` and `provider-match.definition.json`, and that both stay aligned with the `provider-match.ts` contract — so the three copies cannot drift unnoticed.

**Verifier note (confidence: high):** VERIFIED. The anchored exact-membership `where` string at `intake.definition.json:197` and `provider-match.definition.json:73` is byte-identical (confirmed by direct extraction; only outer JSON indentation differs, which is outside the string value). The standalone flow's comment (`provider-match.definition.json:4`) confirms it is orphaned-by-design/dead on the hot path while intake matches inline; both flows comment they mirror `provider-match.ts` (`intake:190`, `provider-match:66`). The duplication is intentional (hot-path inline + ALM-reuse child) and the rule is correct, exactly as stated. `flows/validate-flows.mjs` has NO byte-identity assertion — Check 8a (lines 297-310) only asserts each flow uses some `contains(split(...))` anchored pattern, not that the strings match each other — so the drift hazard is genuinely un-guarded and the recommendation to extend the linter is NOT already-handled. The finding's core is sound.

**Verifier correction:** The finding undercounts the duplication. It says "triplicated (TS + standalone flow + inline intake)" but the identical flow `where` string actually appears in FOUR places: `provider-match.definition.json`, `intake.definition.json`, AND `intake-shared-mailbox.definition.json` (a third flow copy the finding omits) — all three are named in the validator's `PROVIDER_DOMAIN_MATCH_FLOWS` set (`validate-flows.mjs:297-301`) — plus the TS contract. Also, the "TS" copy is NOT a byte-identical string: `provider-match.ts:79-82` is a structurally different `array-.some()/.trim().toLowerCase()` implementation, not the `decodeUriComponent('%0A')` CRLF-strip expression; it is a semantic contract, not a verbatim copy. So the accurate framing is "three byte-identical flow copies + one semantic TS contract," which broadens the drift surface and strengthens the recommendation. Severity is closer to low-medium than medium (all gates off, spike stage, and a green linter+vitest semantic-parity gate already covers the substring/alias regression — just not byte-drift between the literal copies).

## [LOW] Intake's inline provider match folds AMBIGUOUS into the same 'unassigned' branch as no-match, losing the ambiguity signal the standalone flow and TS preserve

**Location:** `flows/definitions/intake.definition.json:202-301` (`If_one_provider` else-branch); contrast `provider-match.definition.json:78-152` (Switch ONE/AMBIGUOUS/NONE) and `mockup-app/src/domain/provider-match.ts:32,89-94`.

**Evidence:** The intake flow branches only on `equals(length(Filter_exact_domain),1)` (`intake.definition.json:204`): exactly-one → bind; everything else (0 OR >1) → `Create_case_unassigned` with a single audit that lumps them together: 'provider_unmatched ... (0 or ambiguous); case unassigned' (`intake.definition.json:287,291`). The canonical implementations distinguish three outcomes — `matched|unmatched|ambiguous` in `provider-match.ts:32` (ambiguous returns the colliding ids, `ts:89-94`) and ONE|AMBIGUOUS|NONE in `provider-match.definition.json:80`. Behaviour is SAFE either way (ambiguous correctly never auto-picks, honouring ADR-0011), but the intake path cannot tell a reviewer 'a domain mapped to >1 active provider, pick one' vs 'no provider at all' — both look like a blank provider. The `15-seed-emaildomains.ps1` ambiguity guard means most intermediary domains are never written, reducing but not eliminating live ambiguity.

**Recommendation:** Optionally split intake's else into AMBIGUOUS vs NONE (mirroring the standalone flow's Switch) so the audit/review surface tells staff WHY the provider is unassigned. Low priority since the no-auto-pick safety property already holds.

**Verifier note (confidence: high):** Verified by reading all three cited files. CONFIRMED: `intake.definition.json:202-204` — `If_one_provider` branches solely on `equals(length(body('Filter_exact_domain')),1)`; the single else branch (lines 256-296) folds 0-match and >1-match together into one `Create_case_unassigned` + one `Audit_provider_unmatched`. The audit name (line 287) is a flat 'provider_unmatched: <domain>' with the count only buried in a free-text `cr1bd_after` string '... N active providers (0 or ambiguous); case unassigned' (line 291); comment at 272 confirms '(0 or ambiguous, or resolve unavailable)'. CONTRAST CONFIRMED: `provider-match.definition.json:78-152` `Switch_match_count` computes ONE|AMBIGUOUS|NONE (line 80) with a distinct `Case_AMBIGUOUS` audit named 'provider_unmatched (ambiguous): <domain>' (line 140) separate from the NONE default 'provider_unmatched' (line 163); `provider-match.ts:32` declares outcome `'matched'|'unmatched'|'ambiguous'` and `ts:89-94` returns `ambiguousProviderIds` (colliding ids). Safety property holds: the else never auto-picks, honoring ADR-0011, so this is an observability/audit-granularity gap, not a correctness bug — correctly self-rated LOW. Important nuance the finding already states accurately: the standalone provider-match flow is ORPHANED-BY-DESIGN (its own header comment, line 4: 'intake currently does the match inline and does NOT call this child... orphaned-by-design'), so the richer three-way audit is not live; only the inline intake path runs. The finding's claim is about the live inline path's own audit conflating the two outcomes, which is true. The `15-seed-emaildomains.ps1` ambiguity guard reduces (not eliminates) live ambiguity, as stated. No overstatement detected; recommendation (optionally split else into AMBIGUOUS vs NONE) is sound and correctly flagged low-priority.
